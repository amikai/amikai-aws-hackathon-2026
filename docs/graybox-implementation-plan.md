# 3D 股伴小屋：灰盒原型實作計畫

> 文件狀態：下一步執行規格  
> 對應文件：`product-concept.md`、`primary-user-journey.md`、`visual-character-direction.md`  
> 目標：先驗證「市場越激動，介面越安定」是否能透過 3D 空間與電子寵物動作被感受到

## 1. 唯一交付物

完成一個可在桌機與手機瀏覽器開啟的 Web 頁面：

- 固定鏡位顯示 3D 小屋。
- 中央有幾何體組成的電子寵物「小伴」。
- 可切換 `CALM` 與 `STORM` 兩種市場狀態。
- `STORM` 時窗外變天、室內暖光增強。
- 小伴依序完成靠近、關窗、陪坐。
- 畫面包含主流程前三格的台詞與選擇。
- 所有市場內容來自單一靜態 Demo JSON。

第一版不追求正式角色外觀，而是回答：

> 即使只使用幾何體，使用者是否能感覺市場在外面，而安全感在小屋裡？

## 3. 技術選擇

建議使用：

- Next.js。
- TypeScript。
- Tailwind CSS。
- React Three Fiber。
- Drei。
- React `useReducer` 管理場景與流程狀態。
- 靜態 JSON 作為唯一資料來源。

第一版不需要：

- 後端 API。
- 資料庫。
- 登入系統。
- Amazon Bedrock。
- Redux 或 Zustand。
- Blender 與正式 GLB 模型。
- 即時行情與 CSV 通用處理管線。

新增套件時使用當下確認過的精確版本，不使用不受控的開放版本範圍。

## 5. 3D 灰盒內容

### 5.1 小屋

使用基礎幾何體：

- `BoxGeometry`：地板、牆壁、書桌、窗框。
- `PlaneGeometry`：窗外背景、路線圖、日記。
- 簡單透明材質或粒子：雨勢。
- `DirectionalLight`：戶外環境光。
- `PointLight`：室內暖琥珀燈。

固定配置：

```text
窗戶             投資路線圖      收音機
左後             後牆中央        右後

                  小伴
                  中央

法人腳印                          書桌日記
左前                              右前
```

### 5.2 小伴

使用幾何體組成：

- 橢圓或膠囊體：身體。
- 圓球：頭部。
- 小圓球：耳朵與尾巴。
- 短膠囊體：手腳。
- 小平面或球體：眼睛。

灰盒只需要輪廓穩定、圓潤、能讀出動作，不製作毛髮、服裝與複雜表情。

## 6. 狀態設計

### 6.1 市場狀態

```ts
type MarketState = "CALM" | "STORM";
```

#### `CALM`

- 戶外為明亮霧藍。
- 室內為柔和奶油光。
- 雨勢關閉或極弱。
- 小伴位於房間中央並平穩呼吸。

#### `STORM`

- 戶外轉為較深的霧藍。
- 雨勢與窗簾擺動增加，但有強度上限。
- 室內暖琥珀燈光增強。
- 小伴依序靠近、關小窗戶、陪坐。
- 畫面一次只呈現一個主要選擇。

### 6.2 體驗狀態

```ts
type ExperienceState =
  | "ARRIVAL"
  | "GROUNDING"
  | "SIT_TOGETHER"
  | "READY_TO_EXPLORE";
```

主要路徑：

```text
ARRIVAL
  → GROUNDING
  → READY_TO_EXPLORE
```

替代路徑：

```text
ARRIVAL
  → SIT_TOGETHER
```

## 7. 前三格流程

### 第一格：抵達

畫面：窗外風雨、小伴平穩呼吸並慢慢靠近。

台詞：

> 「你來了。今天外面的風有點大，我幫你把重要的事情留在桌上了。」

選項：

- 陪我看一下。
- 今天只想坐坐。

### 第二格：安定

若選擇「陪我看一下」：

- 小伴慢慢關小窗戶。
- 室內暖光增強。
- 小伴回到使用者附近。

台詞：

> 「我們不用一次看完。先從一件事情開始就好。」

若選擇「今天只想坐坐」：

- 小伴直接陪坐。
- 不再顯示資料或新問題。

台詞：

> 「沒關係，我們今天不用理解所有事情。坐一下也很好。」

### 第三格：想起計畫

- 牆上路線圖亮起。
- 顯示明天是每月投入日。
- 不繼續實作後續論壇、法人與反思流程。

台詞：

> 「我記得我們原本約好明天一起往前走一步。」

## 8. 執行順序

### Phase 1：靜態構圖

1. 建立 Next.js 頁面與全螢幕 Canvas。
2. 設定固定相機。
3. 放置地板、兩面牆、窗戶、桌子與小伴。
4. 完成桌機與手機基本構圖。

完成標準：第一眼先看到小伴，所有主要物件可辨識。

### Phase 2：情緒環境

1. 建立 `CALM` 與 `STORM` 切換。
2. 改變窗外顏色與雨勢。
3. 改變室內暖光強度。
4. 確認不使用全畫面紅色與閃爍效果。

完成標準：不讀文字也能感覺戶外激動、室內安定。

### Phase 3：灰盒角色動作

1. 實作呼吸。
2. 實作靠近。
3. 實作關窗。
4. 實作陪坐。
5. 加入 `prefers-reduced-motion` 替代效果。

完成標準：動作像在陪伴，不像小伴自己需要被安撫。

### Phase 4：前三格流程

1. 加入台詞面板。
2. 加入兩個抵達選項。
3. 串接主要與陪坐分支。
4. 顯示第三格投資路線圖。
5. 加入「重新開始」開發按鈕。

完成標準：90 秒內可完整走完主要或陪坐路徑。

### Phase 5：裝置檢查

1. 桌機檢查。
2. 手機直向檢查。
3. 低動態模式檢查。
4. 基本鍵盤操作檢查。

## 9. 驗收標準

灰盒完成需同時符合：

1. 有一個可直接開啟的 Web 頁面。
2. 第一眼先看到小伴，而非數字或控制項。
3. 小屋具有固定鏡位，不能自由旋轉。
4. 可手動切換 `CALM` 與 `STORM`。
5. `STORM` 時窗外變化明顯但不製造災難感。
6. `STORM` 時室內暖光增強，不使用全畫面紅色。
7. 小伴能完成呼吸、靠近、關窗與陪坐。
8. 使用者可選擇「陪我看一下」或「今天只想坐坐」。
9. 陪坐分支不再追加問題。
10. 主要分支能到達第三格投資路線圖。
11. 桌機與手機直向都能看見小伴與當前互動。
12. 減少動態效果後流程仍可完成。
13. 所有顯示的市場數字都來自單一 Demo JSON。
14. 90 秒內可走完整個灰盒流程。

## 10. 明確不做

灰盒驗收前不做：

- 正式角色建模與骨骼動畫。
- 完整八格 Storyboard。
- 論壇收音機詳細互動。
- 法人大腳印詳細互動。
- AI 對話生成。
- 後端與資料庫。
- 登入、真實買賣紀錄或券商串接。
- 多標的與多日期。
- 換裝、家具、商城與遊戲化。
- 通用 CSV 匯入與事件計算。
- 正式 Landing Page。

## 11. 完成定義

灰盒不是以程式碼完成度判定，而是以以下問題判定：

> 在沒有正式模型、完整資料與 AI 的情況下，使用者是否已經能感覺到小伴在市場風暴中提供一個溫暖、安定且不催促的空間？

若答案仍不明確，優先調整鏡位、光線、節奏與角色動作，不增加新功能。
