# LLM 體驗策略：讓 AI 藏在陪伴背後

> 文件狀態：MVP 產品原則  
> 對應文件：`product-concept.md`、`primary-user-journey.md`、`graybox-implementation-plan.md`  
> 核心結論：介面不是聊天室，互動不是 Prompt，LLM 也不是主角

## 1. 問題

「使用者輸入一句話、AI 回覆一段文字」的產品，通常只會吸引少數願意主動提問、整理需求與閱讀長答案的重度使用者。

大盤 ETF 定期投資新手通常：

- 不知道應該問什麼。
- 不想每天輸入問題。
- 市場高波動時沒有餘裕閱讀長分析。
- 需要的是被整理好的情境，而不是更多資訊。
- 更容易透過固定儀式與低摩擦互動形成習慣。

因此，股伴不能是披著電子寵物外觀的聊天機器人。

## 2. LLM 的唯一定位

> 我們不是讓使用者和 LLM 對話，而是讓 LLM 讓電子寵物知道此刻應該怎麼陪伴。

LLM 是藏在角色背後的個人化陪伴引擎：

- **翻譯者**：把已確認的資料事實轉成適合此刻的一句話。
- **記憶整理者**：把使用者的選擇整理成可確認的共同日記。
- **回顧敘事者**：把一段時間的互動整理成簡短回顧。

LLM 不是：

- 股價預測者。
- 投資分析師。
- 買賣建議引擎。
- 市場事件計算器。
- 使用者情緒診斷工具。
- 3D 動畫控制器。
- 主畫面的聊天介面。

## 3. Zero-input first

核心流程必須在完全不輸入文字的情況下成立：

```text
打開小屋
→ 小伴主動察覺市場狀態
→ 使用者點一下
→ 探索一件事
→ 做一個選擇
→ 小伴整理共同記憶
→ 安心離開
```

產品不能以以下問題開場：

> 「今天有什麼想問的嗎？」

應該先替使用者整理情境：

> 「你來了。今天外面的風有點大，我幫你把重要的事情留在桌上了。」

### 3.1 輸入的三個層級

#### Level 0：不用輸入

系統已經知道：

- 日期與市場狀態。
- 使用者原定投入日。
- 已確認的價格、論壇與法人事實。
- 過去明確保存的計畫與偏好。

小屋與小伴先根據這些狀態做出反應。

#### Level 1：動作與選項

主要互動是：

- 點擊小伴。
- 點擊收音機、腳印或路線圖。
- 選擇「有點擔心」。
- 選擇「陪我看一下」。
- 選擇「今天只想坐坐」。
- 選擇「明天再確認」。

每個畫面最多提供二至四個選項。

#### Level 2：自由文字，永遠選填

只在共同日記結尾提供：

> 想留一句話給未來的自己嗎？（選填）

不輸入文字仍可完成全部流程。自由文字是加深體驗的入口，不是使用產品的門檻。

## 4. 互動主體是小屋與電子寵物

使用者看到的是：

- 小伴如何靠近與陪坐。
- 小伴何時關窗、觀察或保持沉默。
- 今天只有哪一件事值得理解。
- 過去的選擇是否被正確記得。
- 日記是否像自己的經歷。

使用者不應主要看到：

- 大型文字輸入框。
- 「問小伴任何問題」。
- AI 正在輸入的聊天室。
- 無限多輪追問。
- 長篇生成式投資分析。
- 必須撰寫的每日心得。

## 5. MVP 的兩個 LLM 能力

### 5.1 微型個人化台詞

輸入由結構化狀態組成：

```json
{
  "market": "high_volatility",
  "nextInvestment": "tomorrow",
  "planHorizon": "10_years",
  "pastBehavior": "paused_once_after_drawdown",
  "preferredExplanation": "short",
  "currentChoice": "worried"
}
```

LLM 只產生一個短回應：

> 「我記得上次遇到大風時，你也擔心過。明天是原本約好的日子，我們今天不用先做決定。」

價值在於處理「市場狀態 × 計畫 × 過去行為 × 偏好 × 當前選擇」的組合，而不是提供新市場判斷。

### 5.2 共同日記草稿

即使使用者沒有打字，也可以根據點擊與選擇產生：

```text
今天市場波動很大。
小安感到擔心，選擇先記錄，明天再確認原本計畫。
```

保存前必須讓使用者：

- 確認。
- 編輯。
- 取消。
- 選擇不保存。

## 6. 後續能力

不屬於第一版必要範圍：

### 6.1 月度回顧

把一個月的結構化互動整理成短故事：

> 「這個月你完成了原定投入，也曾在市場波動時想暫停。你沒有急著替自己評分，而是把擔心記下來，隔天重新確認計畫。」

### 6.2 個人化知識說法

同一事實依偏好提供不同深度：

簡短版：

> 「外面確實很吵，但大部分討論沒有明確方向。」

數字版：

> 「今天共有 196 則發文，其中 121 則是中性，明確看空只有 10 則。聲量增加不等於看空共識。」

知識主題由確定性規則選擇，LLM 只調整說法。

### 6.3 選填自由文字理解

若使用者選擇輸入：

> 「最近跌好多，我也怕公司裁員，所以想停一下。」

LLM 可以拆分市場焦慮與生活需求，但不能診斷或替使用者決定：

> 「我聽見最近的下跌讓你不安，也擔心工作變化會影響生活。後者可能真的會改變原本計畫，我們可以先把這兩件事分開整理。」

## 7. 回應長度與語言規則

每次回應最多包含：

1. 一句承接。
2. 一句事實。
3. 一個邀請。

建議限制在 40–70 個中文字，詳細數字由使用者點擊抽屜查看。

語言遵循：

> 同理 → 事實 → 邀請

禁止：

- 預測未來。
- 使用「應該買入／賣出」。
- 將維持計畫描述成唯一正確答案。
- 推論未經使用者表達的情緒。
- 用長篇文字壓過 3D 寵物與場景。
- 用幼兒化語言稱讚成年人。

## 8. 確定性程式與 LLM 的分工

### 8.1 確定性程式負責

- 計算報酬、法人力道與論壇異常。
- 決定市場激動程度。
- 選取正確的原始數字。
- 選擇今天是否有事件。
- 控制 3D 場景與動畫。
- 管理使用者明確設定的計畫。
- 驗證顯示內容與資料來源。

### 8.2 LLM 負責

- 依結構化脈絡生成短句。
- 將已發生的互動整理成日記草稿。
- 依使用者偏好調整知識說法。
- 未來生成月度回顧。

流程：

```text
CSV 與使用者計畫
  ↓
確定性資料計算
  ↓
結構化市場事實與互動狀態
  ↓
LLM 生成短句與日記草稿
  ↓
規則與格式驗證
  ↓
小伴說出來／使用者確認保存
```

## 9. LLM 不直接控制 3D 動畫

體驗狀態由前端狀態機控制：

```text
APPROACH
GROUND_AND_SIT
INSPECT_CLUE
WRITE_DIARY
REST
```

第一版 LLM 只生成文字與記憶草稿。即使未來回傳 `suggestedAction`，前端也只能接受白名單動作；未知動作一律忽略。

這能避免：

- 動作與畫面節奏失控。
- LLM 讓寵物表現恐慌或責怪。
- 回應延遲阻塞抵達與安定流程。

## 10. 延遲與失敗降級

- 抵達、關窗與陪坐完全不等待 LLM。
- 第一段台詞可使用已審核模板立即顯示。
- LLM 僅在需要個人化短句或生成日記時呼叫。
- 呼叫失敗時退回固定模板。
- LLM 失敗不能阻止使用者完成流程或離開小屋。
- 不在畫面上以焦慮的「AI 正在思考」等待使用者。

## 11. 輸出格式

建議採固定 JSON：

```json
{
  "acknowledgement": "最近的變化確實很大，感到擔心很正常。",
  "observation": "今天論壇聲量增加，但多數發文沒有明確方向。",
  "invitation": "你想看看腳印，還是今天先坐一下？",
  "memoryDraft": "小安在投入日前因市場波動感到擔心，選擇先觀察。",
  "factsUsed": ["forum_posts", "forum_neutral", "next_investment_date"]
}
```

輸出必須驗證：

- JSON 結構正確。
- `factsUsed` 都存在於輸入。
- 沒有新增未提供的數字。
- 沒有買賣指令與未來預測。
- 文字符合長度限制。

## 12. 大眾使用者的回訪來自儀式

一般使用者不會因為想和 AI 聊天而回來，而會因為產品承接了固定生活時刻：

### 投入日前

> 「明天是我們原本約好的日子，要一起確認一下嗎？」

### 市場高波動時

> 「外面很吵，我把重要的事情整理在桌上了。」

### 投入完成後

> 「這一步已經記進我們的路線了。」

### 月底

> 「這個月有兩次讓你想改變計畫的時刻，要一起回顧嗎？」

LLM 的作用是讓相同儀式能持續貼近這個人的計畫與記憶，而不是製造更多聊天時間。

## 13. 提案說法

不使用：

> 「使用者可以和 AI 寵物聊天、詢問投資問題。」

改用：

> 「股伴不要求使用者學會如何問 AI。市場狀態、投資計畫與過去行為已構成上下文；使用者只需要和電子寵物及小屋互動，LLM 在背後把脈絡轉成一句適合此刻的陪伴。」

提案主句：

> **我們不用 AI 預測市場，而是讓 AI 理解此刻該如何陪伴一個人。**

更完整的定位：

> 資料告訴我們發生什麼，確定性程式控制可信的體驗，LLM 讓陪伴變得更懂這個人，3D 電子寵物則用溫暖、低摩擦的方式把陪伴表達出來。

## 14. MVP 驗收條件

1. 完全不輸入文字也能走完流程。
2. 主畫面沒有大型文字輸入框。
3. 不出現「問小伴任何問題」。
4. 每個畫面最多二至四個選項。
5. LLM 每次只輸出短句與日記草稿。
6. LLM 回應不阻塞 3D 動畫與流程。
7. LLM 失敗時可退回模板並完成體驗。
8. 事件、數字、動畫與買賣決策不由 LLM 控制。
9. 日記保存前必須由使用者確認。
10. 使用者不需要知道 Prompt，也不需要理解 LLM 才能使用產品。
